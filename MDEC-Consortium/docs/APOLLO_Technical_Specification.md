# APOLLO Technical Specification (ATS-01)
## Reference Implementation: The Metadata Muse

**Version:** 1.0.0 (Draft)  
**Date:** January 15, 2026  
**Status:** PROTOTYPE VERIFIED  
**Parent Standard:** MDEC-CODEX

---

## 1. System Overview
APOLLO is the official "Reference Viewer" for MDEC-compliant archives. It is defined as a **Lightweight, Client-Side Metadata Interaction Layer**.

### 1.1 Core Tenets
1.  **Zero-Heavy-Infrastructure:** Must run on minimal resources (Static HTML/JS + optional lightweight local server).
2.  **Universal Portability:** The Viewer and the Data travel together (Portable Vaults).
3.  **Visual Governance:** Metadata quality is assessed visually via the "MOSART-ID-Card".

---

## 2. Architecture

### 2.1 The Data Layer (The Hive)
*   **Format:** JSON (Standardized Schema).
*   **Source:** `archive_metadata.json` (Generated by MDEC Mining Tools).
*   **Schema Definition:**
    ```json
    {
      "FileName": "String (Required)",
      "Path": "String (Relative to Vault Root)",
      "Category": "String (MDEC Standard Category)",
      "Tags": ["Array", "of", "Strings"],
      "Checking algorithms": "SHA256",
      "Description": "Extracted Summary or Frontmatter",
      "MDEC_Score": "Integer (0-100)"
    }
    ```

### 2.2 The Presentation Layer (The Muse)
*   **Technology Stack:** HTML5, CSS3 (CSS Variables for Theming), Vanilla JavaScript (ES6+).
*   **No Build Step:** The source code is the executable code.
*   **Design Language:** "Neural Night" (Dark Mode, Neon Accents, Card-Based UI).

### 2.3 The Logic Layer (The Brain)
*   **Client-Side Indexing:** The JSON payload is loaded into browser memory for instant search/filtering (supporting up to ~100k records comfortably).
*   **Search Algorithm:** Fuzzy matching on `FileName`, `Tags`, and `Category`.

---

## 3. Interface Specification: The MOSART-ID-Card

The **MOSART-ID-Card** is the atomic unit of the Apollo interface.

| Component | specification | Visual Cues |
| :--- | :--- | :--- |
| **Header** | Asset Name + Ext | Bold Typography |
| **Badge** | Category | Color-Coded (Green=Doc, Blue=Code, etc.) |
| **Hologram** | Checksum/ID | Monospace/Techno font |
| **Tags** | Taxonomy Keywords | Pill-shaped, semi-transparent backgrounds |
| **Health** | MDEC Score | Green Border (Good) vs Red Border (Needs Fix) |

---

## 4. Future Roadmap: Read/Write Capabilities

To achieve "Universal Reader/Writer" status, the following modules are planned:

### 4.1 Phase 2: The "Editor" (Q1 2026)
*   **Feature:** In-Place Editing.
*   **Mechanism:**
    *   User edits a tag in the HTML Card.
    *   Change is queued in a `changes.json` delta file.
    *   PowerShell/Python script applies deltas to source files (`.md` frontmatter).

### 4.2 Phase 3: The "Extension" (Q2 2026)
*   **Feature:** VS Code Integration.
*   **Mechanism:**
    *   `HoverProvider` reading the local `archive_metadata.json` index.
    *   Renders the MOSART-ID-Card directly in the IDE.
